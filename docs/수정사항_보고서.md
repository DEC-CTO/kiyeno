# 기능개발 수정사항 보고서

## 1. 신규자재 추가 시 사이즈 입력값 저장 안 됨

| 항목 | 내용 |
|------|------|
| **담당자** | 금정훈 |
| **상태** | 기완료 |
| **피드백** | 신규자재 추가 시 사이즈 입력해도 저장이 안됨 |

### 증상
- 경량부품 추가 화면에서 사이즈 필드에 값을 입력하고 저장해도 해당 값이 저장되지 않는 문제

### 원인
- `priceDatabase.js`의 `addLightweightComponent()` 함수에서 새로운 자재 객체를 생성할 때 `size` 필드가 누락되어 있었음
- 입력 단계(`app-services.js`)에서는 사이즈 값을 정상적으로 수집하고 있었으나, 저장 단계에서 해당 필드를 객체에 포함하지 않아 값이 소실됨

### 수정 내용
- `priceDatabase.js`의 `addLightweightComponent()` 함수 내 자재 객체 생성부에 `size: materialData.size || ''` 항목 추가
- 공종1/공종2/부위 필드의 하드코딩 기본값(`'경량'`, `'벽체'`) 제거 → 사용자 입력값 존중하도록 변경

## 2. 자재관리 초기화 해도 기본 데이터베이스가 안 지워짐

| 항목 | 내용 |
|------|------|
| **담당자** | 금정훈 |
| **프로젝트** | 대만 T3 프로젝트 |
| **상태** | 완료 (접수→가능→확정→완료) |
| **피드백** | 자재관리 초기화 해도 기본 데이터베이스가 안 지워짐 |
| **비고** | 현재 상태가 초기화 상태로 협의됨. 변경 필요 시 전체 구성원 합의 필요 |

### 증상
- 자재 관리 화면에서 초기화 기능을 실행해도 기본 데이터베이스(하드코딩된 경량자재/석고보드)가 삭제되지 않는 문제

### 원인
- 기존 초기화 기능이 사용자 추가 데이터만 제거하고, 하드코딩된 기본 데이터(55개 경량자재 + 49개 석고보드)는 항상 복구하는 구조였음

### 수정 내용
- 초기화 옵션을 2가지로 분리하여 드롭다운 메뉴에 제공
  - **원본으로 초기화**: 모든 수정사항을 삭제하고 기본 데이터로 복구 (기존 동작)
  - **완전초기화**: 기본 데이터 포함 모든 자재 데이터를 삭제하여 빈 테이블 상태로 만듦
- `priceDatabase.js`에 `_isFullReset` 플래그 및 `localStorage` 상태 관리를 추가하여, 완전초기화 후 페이지 새로고침 시에도 기본 데이터가 자동 복구되지 않도록 처리
- `app-services.js`에 `resetToOriginal()`, `fullResetToEmpty()` 함수 및 확인 경고 모달 추가
- 수정 파일: `app-services.js`, `priceDatabase.js`, `app-ui.js`

## 3. 월타입 신규 작성시 시간별 저장만 가능, 분류 정렬이 안됨

| 항목 | 내용 |
|------|------|
| **담당자** | 박미전 |
| **프로젝트** | 대만 T3 프로젝트 |
| **상태** | 완료 (접수→가능→확정→완료) |
| **피드백** | 월타입 신규 작성시, 시간별 저장만 가능, 분류 정렬이 안됨 |
| **비고** | 전체 구성원 협의 후 드래그 드랍 방식으로 결정 |

### 증상
- 벽체 타입 목록에서 순서를 변경할 수 없고, 시간순(저장순)으로만 표시됨
- 사용자가 원하는 순서로 분류/정렬하는 기능이 없음

### 원인
- 벽체 타입 목록에 순서 정보(`sortOrder`) 필드가 없어 저장 시간순으로만 표시되었음
- 사용자 정의 정렬 기능이 구현되어 있지 않았음

### 수정 내용
- 3개 모듈에 드래그 드랍 순서 변경 기능 추가
  - **excelWallTypeManager.js**: Excel 일위대가 벽체타입 목록에 드래그 드랍 구현, `sortOrder` 필드로 순서 저장 (IndexedDB)
  - **revitTypeMatching.js**: Revit 벽체 타입 매칭 목록에 드래그 드랍 구현, 기존 알파벳 정렬을 `sortOrder` 기반 정렬로 변경
  - **unitPriceManager.js**: 일위대가 구성품 행에 드래그 드랍 구현, DOM 순서 변경 후 합계 재계산
- No(번호) 컬럼을 드래그 핸들로 지정 (cursor: grab 스타일, 툴팁 표시)
- 드래그 시 시각적 피드백 제공 (투명도 변경, 파란 테두리 표시)

## 4. 키비스에서 레빗 연동시 잦은 오류 발생 (한번 실패시 다시 서버를 나갔다 들어와야 하는 불편함)

| 항목 | 내용 |
|------|------|
| **담당자** | 능산유 |
| **프로젝트** | 대만 T3 프로젝트 |
| **상태** | 기완료 |
| **피드백** | 키비스에서 레빗 연동시 잦은 오류 발생 (한번 실패시 다시 서버를 나갔다 들어와야 하는 불편함) |

### 증상
- 레빗 연동 중 오류 발생 시 페이지가 먹통이 되어 Ctrl+F5(강제 새로고침)를 해야만 복구됨
- 장시간 사용(30분~3시간) 후 버튼이 먹통이 되는 현상 발생
- 싱글 페이지 애플리케이션 특성상 메모리 누수가 누적되어 UI가 응답하지 않게 됨

### 원인
- 싱글 페이지 애플리케이션(SPA) 구조에서 자주 발생하는 메모리 누수 문제로, 페이지 전환 없이 장시간 사용 시 이벤트 리스너와 타이머가 누적되어 성능이 저하됨
- 9개의 이벤트 리스너 메모리 누수 발견 (중복 등록, 미해제 등)
- 자동 저장 타이머(30초 간격)가 페이지 닫아도 계속 실행됨
- WebSocket 통신 지연 및 대량 데이터 직렬화/역직렬화 오버헤드
- 전역 에러 핸들러 부재로 처리되지 않은 에러가 앱 전체를 중단시킴

### 수정 내용
- **메모리 누수 방지**
  - 9개 이벤트 리스너 중복 제거 (`unitPriceManager.js`, `app-ui.js`, `app-services.js`, `revitTypeMatching.js`, `wall-cost-calculator.js`)
  - 자동 저장 타이머 정리: `beforeunload` 이벤트에서 `clearInterval` 처리 (`app-core.js`)
  - 모달 keydown 리스너 누적 방지, 드롭다운 click 리스너 동적 등록 방식으로 변경
- **에러 처리 강화**
  - 전역 에러 핸들러 추가: `window.error`, `unhandledrejection` 처리 (`main.js`)
  - try-catch 블록 96% 이상 커버리지 확보
  - 계산 실패 시 토스트 알림으로 사용자 피드백 제공 (`wall-cost-calculator.js`)
- **모니터링 시스템 추가**
  - 1분마다 메모리 사용량 체크, 90% 이상 시 경고 (`main.js`)
  - 1시간마다 장시간 사용 감지, 2시간 이상 사용 시 새로고침 권장

## 5. 노무비 입력 및 저장 문제

| 항목 | 내용 |
|------|------|
| **담당자** | 정재호 |
| **프로젝트** | 대만 T3 프로젝트 |
| **구분** | 자재관리 |
| **상태** | 기완료 |
| **피드백** | 1. 새로운 공종 입력시 노무비를 100단위 기입해야만 저장 가능하며, 경고메시지도 없어 뭐가 잘못되었는지 알수없음 / 2. 노무비 수기로 입력가능했으면 좋겠습니다 |

### 증상
- 경량부품 편집 시 노무비를 100단위로 기입해야만 저장 가능하고, 경고 메시지가 없어 원인 파악 불가
  - 예: 노무비 100, 기준노무비 100 입력 → 최종 노무비 1원으로 잘못 계산됨
- 노무비를 수기로 직접 입력하는 기능이 없음

### 원인
- 기존 노무비 처리 로직이 일반 자재와 동일한 방식(단가 × 수량 = 금액)으로 계산되어, 노무비 특성에 맞지 않는 계산 결과가 발생
- 노무비 금액을 단가 필드에 입력하는 구조로 되어 있어, 수량으로 나눠지면서 의도하지 않은 값이 산출됨

### 수정 내용
- **노무비 계산 방식 변경**: 노무비는 일반 자재(단가×수량=금액)와 반대로, 금액을 직접 입력하고 단가는 금액÷수량으로 자동 계산되도록 변경
- **노무비 항목 자동 판별**: `isLaborCost()` 함수 추가로 노무비/인건비/시공비 등 키워드 기반 자동 판별
- **노무비 수기 입력 지원**: 자재 선택 시 노무비 데이터가 있으면 금액 컬럼에 직접 값을 설정하고, 수량 변경 시 단가가 자동 재계산됨
- **노무비 금액 표시 위치 정정**: 단가 컬럼에는 계산된 단가, 금액 컬럼에는 입력된 금액이 올바르게 표시되도록 수정
- 수정 파일: `unitPriceManager.js`

## 6. 벽체관리에서 자재삽입 순서 변경 기능 요청

| 항목 | 내용 |
|------|------|
| **담당자** | 정재호 |
| **프로젝트** | 대만 T3 프로젝트 |
| **구분** | 벽체관리 |
| **상태** | 완료 (접수→기술검토중→완료) |

- 드롭 후 `calculateGrandTotal()` 호출로 재료비/노무비 합계 자동 재계산

## 7. 벽체 타입 관리 테이블 두께 컬럼 위치 이동

| 항목 | 내용 |
|------|------|
| **상태** | 완료 (접수→가능→완료) |
| **피드백** | 맨뒤에 있는 벽체 두께를 월타입 뒤쪽으로 위치 이동 |

### 증상
- 벽체 타입 관리 테이블에서 "두께(mm)" 컬럼이 테이블 맨 뒤(옵션 뒤, 자재비 앞)에 위치하여 WallType과의 연관성을 한눈에 파악하기 어려움

### 원인
- 초기 테이블 설계 시 두께 컬럼이 마감/구조체/단열재 컬럼 뒤에 배치되어 있었음

### 수정 내용
- "두께(mm)" 컬럼을 WallType 컬럼 바로 뒤로 위치 이동
  - 변경 전: 체크박스 → No → WallType → 좌측마감 → 구조체 → 단열재 → 우측마감 → 옵션 → **두께** → ...
  - 변경 후: 체크박스 → No → WallType → **두께** → 좌측마감 → 구조체 → 단열재 → 우측마감 → 옵션 → ...
- 벽체 타입 관리(`excelWallTypeManager.js`)와 Revit 타입 매칭(`revitTypeMatching.js`) 양쪽 테이블 모두 동일하게 적용
- 수정 파일: `excelWallTypeManager.js`, `revitTypeMatching.js`, `revit-wall-handler.js`, `app-ui.js`

## 8. 일위대가 C-50형 아이템 벽체 타입관리 적용 시 석고보드 450으로 표시되는 오류

| 항목 | 내용 |
|------|------|
| **상태** | 완료 (접수→가능→완료) |
| **피드백** | 일위대가 C-50형 아이템이 벽체 타입관리 적용시 석고보드 450이라고 뜨는 오류 |
| **비고** | 사진대지 포함 |

### 증상
- 벽체 타입 관리의 구조체 컬럼에 일위대가를 적용했을 때, `unitPrice_석고보드-450-3600이하-50 정-1759287113390` 같은 원본 ID가 그대로 표시됨
- C-50형 아이템인데 석고보드 450으로 잘못 표시되는 문제

### 원인
- 일위대가 ID를 표시명으로 변환하는 `getUnitPriceDisplayName()` 함수에서 캐시 조회 실패 시 원본 ID를 그대로 반환하는 구조였음
- 일위대가 데이터를 메모리 배열, localStorage, IndexedDB 3가지 방식으로 혼용하여 로드하면서 캐시 동기화가 안 되는 문제 발생

### 수정 내용
- **단일 데이터 소스 도입**: `revitTypeMatching.js`에서 일위대가 데이터 로드를 `window.getAllUnitPricesForExternal()` 단일 함수로 통일하여 메모리-DB 불일치 문제 해결
- **DB 직접 조회**: `unitPriceManager.js`에서 메모리 캐시 의존 대신 IndexedDB에서 직접 조회하도록 변경
- **두께 자동 재계산**: 일위대가 적용 직후 `recalcWallThickness()` 함수를 호출하여 두께가 자동으로 재계산되도록 추가
- 수정 파일: `revitTypeMatching.js`, `unitPriceManager.js`, `excelWallTypeManager.js`

## 9. 벽체 타입 작성 시 레이어 컬럼 자유 추가/삭제 기능 요청

| 항목 | 내용 |
|------|------|
| **상태** | 완료 (접수→가능→완료) |
| **피드백** | 벽체 타입 작성 시 레이어에 한계가 있음 자유자재로 늘리고 줄이고 가능하는게 좋을거 같음 (구조체/단열재 포함) |

## 11. 월 타입 전송 시 CompoundStructure 오류 발생

| 항목 | 내용 |
|------|------|
| **상태** | 완료 (접수→가능→완료) |
| **피드백** | 월 타입 전송 시 오류 발생 구간 - CompoundStructure is not valid 에러 (성공 17개, 실패 12개) |
| **비고** | 접수→개발자확인중 |

### 증상
- Revit으로 벽체 타입 전송(CREATE_WALL_TYPES) 시 다수 벽체에서 "생성 중 오류 발생, 복합 구조 수정 실패" 에러 발생
- 에러 메시지: `CompoundStructure is not valid. Parameter name: compoundStructure`
- 실패 벽체: W-03, W-04, W-08, W-09, W-10, W-13, W-19, W-21, W-22, W-24, W-25, W-32 등 12개

### 원인
- C# Revit 애드인의 `SetStructuralCore()` 메서드에서 `SetNumberOfShellLayers()` API 호출 시, 계산된 외부/내부 쉘 레이어 개수가 실제 레이어 구조와 맞지 않아 CompoundStructure 검증(IsValid) 실패
- 특히 다층 구조(레이어가 많은) 벽체에서 빈번히 발생

### 수정 내용
- **SetStructuralCore 호출 비활성화**: `WallTypeCreate.cs`의 `ModifyToComplexStructure()` 함수에서 `SetStructuralCore()` 메서드 호출을 주석 처리하여 CompoundStructure 검증 오류 해결
- **트랜잭션 개별 처리**: `QTOForm.cs`에서 기존 전체 단일 트랜잭션을 벽체 타입별 개별 트랜잭션으로 변경하여, 한 개의 실패가 다른 벽체 생성에 영향 주지 않도록 격리
- **결과 요약 표시**: 생성 완료 후 성공/실패 개수 및 실패 벽체별 상세 오류 메시지를 TaskDialog로 표시
- **WebSocket 메시지 처리 개선**: 큰 메시지가 분할 전송될 때 `StringBuilder`로 조각을 누적하여 완전한 메시지 수신 후 처리하도록 변경
- 수정 파일: `QTO/QTO/WallTypeCreate.cs`, `QTO/QTO/QTOForm.cs` (C# Revit 애드인)

## 12. CAD 벽체 시공색 Revit 적용 및 벽체별 색상 구분 기능

| 항목 | 내용 |
|------|------|
| **상태** | 기완료 / 기구현완료 |
| **피드백** | 1. 캐드파일에서 불러온 벽체 시공색이 바로 데빗에 적용되면 좋을까 같음 / 2. 벽체마다 색상으로 구분되면 좋을까같음 |

### 증상
- CAD 파일에서 불러온 벽체의 시공색이 Revit에 자동 적용되지 않음
- 벽체 타입별 시각적 구분이 어려워 작업 효율 저하

### 원인
- Revit 색상 오버라이드 기능이 구현되어 있지 않았음
- 웹 테이블 및 Revit 3D 뷰에서 벽체별 색상 구분 기능이 없었음

### 수정 내용
- **웹 테이블 벽체 색상 구분**: `revit-wall-handler.js`에서 벽체 타입별 고유 색상을 자동 생성(HSL 컬러 스페이스 기반)하여 테이블 행에 배경색 적용 (25% 투명도 + 좌측 4px 색상 바)
- **Revit 3D 뷰 색상 오버라이드**: `QTOForm.cs`의 `ApplyWallColors()` 함수에서 표면(Surface)과 절단면(Cut) 모두에 색상 적용, 평면도/입면도/단면도에서도 색상 표시
- **색상표(범례) 생성**: Revit에서 드래프팅 뷰로 색상표 자동 생성 (벽체 타입명 + 색상 박스 + 개수 표시)
- **색상 초기화**: 적용된 색상을 원래 상태로 되돌리는 기능
- **자재별 벽체 색상 표시**: `priceComparisonManager.js`에서 특정 자재가 사용된 벽체를 Revit 3D 뷰에서 색상으로 하이라이트
- UI: 드롭다운 메뉴에 "벽체 색상 반영", "색상표 생성", "색상 초기화" 버튼 추가
- 수정 파일: `revit-wall-handler.js`, `styles.css`, `index.html`, `QTOForm.cs`, `priceComparisonManager.js`

## 18. 벽체타입 생성 미리보기 시 두께 추출 실패 오류

## 19. 벽체 색상 반영 시 평면도/단면도에 적용 안 됨

| 항목 | 내용 |
|------|------|
| **상태** | 완료 (접수→가능→완료) |
| **피드백** | 색상반영 → 평면 안됨 (절단 패턴 동일하게 적용하면 어떨지?) → 단면부분도 적용되게 변경 완료 |

### 증상
- "벽체 색상 반영" 기능 실행 시 3D 뷰와 입면도에서는 색상이 정상 표시되나, 평면도 및 단면도에서는 색상이 적용되지 않음

### 원인
- Revit에서 평면도와 단면도는 벽체의 절단면(Cut) 표현을 사용하는데, 기존 구현에서 표면(Surface) 패턴 색상만 설정하고 절단면(Cut) 패턴 색상은 설정하지 않았음

### 수정 내용
- `ApplyWallColors()` 함수에서 표면(Surface) 색상 설정에 추가로 절단면(Cut) 색상 설정 코드 추가
  - `SetCutForegroundPatternColor()`, `SetCutBackgroundPatternColor()`: 절단면 색상 적용
  - `SetCutForegroundPatternId()`, `SetCutBackgroundPatternId()`: 절단면에 솔리드 채우기 패턴 적용
- 이로써 3D 뷰, 입면도, 평면도, 단면도 모든 뷰 타입에서 벽체 색상이 표시됨
- 수정 파일: `QTOForm.cs`

## 20. 벽체 색상 반영/초기화 시 선택 및 필터 동작 개선

| 항목 | 내용 |
|------|------|
| **상태** | 완료 (접수→가능→완료) |
| **피드백** | 1. 벽을 선택하지 않아도 색상 반영이 됨 2. 필터로만 색상 반영 됨 → 선택한 객체만 색상 반영이 안됨 3. 필터를 적용시키면 색상초기화가 안됨 |

### 증상
- 체크박스로 벽체를 선택하지 않아도 전체 벽체에 색상이 반영됨
- 필터 적용 후 체크박스로 선택한 객체만 색상을 반영할 수 없음
- 필터를 적용/초기화하면 기존에 적용된 색상이 사라지고, 색상 초기화도 정상 동작하지 않음

### 원인
- `applyWallColors()` 함수가 체크박스 선택 상태를 확인하지 않고 필터된 전체 벽체에 색상을 적용하는 구조였음
- 필터 적용/초기화 시 테이블이 다시 렌더링되면서 기존 색상 정보가 유실됨

### 수정 내용
- **선택한 벽체만 색상 반영**: `applyWallColors()` 함수에서 `.revit-row-checkbox:checked` 체크박스 상태를 확인하여 선택된 벽체만 색상 적용, 미선택 시 경고 메시지 표시
- **필터 후 선택한 벽체만 색상 반영**: 필터 적용 후에도 체크박스 선택 기반으로 동작하도록 변경
- **필터 시 색상 유지**: `reapplyColorsToTable()` 함수를 추가하여 필터 적용/초기화 후 `wallColorMap`에 저장된 기존 색상을 ElementId 기준으로 재적용
- **선택적 색상 초기화**: `clearWallColors()` 함수도 체크박스 선택된 벽체만 초기화하도록 변경, 미선택 벽체의 색상은 유지
- 수정 파일: `revit-wall-handler.js`

## 21. 자재 관리 테이블 싸이즈 정보 수정

| **피드백** | 선택된 벽체만 데이터 삭제? 이 기능이 필요할지 모르겠지만 계산하기 할때 선택을 하지 않고 계산하면 가능 함 → 선택한 벽체만 삭제되는 기능으로 변경함 |

### 증상
- "데이터 지우기" 버튼 클릭 시 전체 Revit 벽체 데이터가 일괄 삭제됨
- 확인 메시지: "정말로 32개의 Revit 벽체 데이터를 모두 삭제하시겠습니까?"로 전체 삭제만 가능
- 특정 벽체만 선택적으로 삭제할 수 없음

### 원인
- `clearRevitData()` 함수가 `revitWallData = []`로 전체 배열을 초기화하는 구조여서 선택적 삭제가 불가능했음

### 수정 내용
- 체크박스 선택 기반 삭제로 변경: `.revit-row-checkbox:checked`로 선택된 행만 대상으로 삭제
- 선택된 벽체의 ElementId를 `Set`으로 수집 후, `Array.filter()`로 해당 항목만 제거
- 확인 메시지도 선택된 개수 기반으로 변경 (예: "3개의 선택된 벽체 데이터를 삭제하시겠습니까?")
- 미선택 시 "삭제할 벽체를 선택하세요" 안내 표시
- 삭제 후 필터 재적용(`applyRevitFilters()`)으로 테이블 정상 갱신
- 수정 파일: `revit-wall-handler.js`

## 28. 색상범례(색상표) 생성 기능 추가

| 항목 | 내용 |
|------|------|
| **상태** | 완료 (접수→가능→완료) |
| **피드백** | 색상범례기능 추가 요청. 디테일 뷰로 색상 적용 1:50 기준으로 생성. 임의로 생성된 색상범례를 저장하고, 저장된 색상범례를 레빗의 범례로 저장 |

### 증상
- "벽체 색상 반영" 후 어떤 색상이 어떤 벽체 타입을 의미하는지 확인할 수 있는 범례가 없음
- 현장에서 색상이 적용된 도면을 볼 때 벽체 타입 구분이 어려움

### 원인
- 벽체 색상 반영 기능은 있었으나, 색상과 벽체 타입의 대응 관계를 시각적으로 표시하는 범례 생성 기능이 구현되어 있지 않았음

### 수정 내용
- **웹 측**: "색상표 생성" 버튼 추가, `createLegendView()` 함수에서 `wallColorMap`의 색상 데이터를 LegendItem 배열로 변환하여 Revit으로 WebSocket 전송
- **Revit 측**: `CreateLegendView()` 함수에서 Drafting View(제도뷰)를 1:50 스케일로 생성
  - `FilledRegion`으로 색상 박스 생성 (각 벽체 타입별 RGB 색상 적용)
  - `TextNote`로 벽체 타입명 + 개수 레이블 표시 (예: "SH-03 (5개)")
  - 색상별 `FilledRegionType`을 `QTO_Color_R_G_B` 이름으로 자동 생성
- 기존 동일 이름의 범례뷰가 있으면 삭제 후 재생성
- 생성 완료 후 해당 제도뷰를 자동 활성화
- 수정 파일: `revit-wall-handler.js`, `QTOForm.cs`, `index.html`

## 30. CAD 벽체 생성 시 시계방향 정방향 적용 (벽 중심선 기준)

| 항목 | 내용 |
|------|------|
| **상태** | 완료 (접수→가능→완료) |
| **피드백** | 시계방향 정 방향으로 적용 : 레빗과 동일하게 |

### 증상
- CAD 도면에서 Revit으로 벽체를 생성할 때 벽의 위치 기준선이 일관되지 않아 벽체가 의도한 위치에 정확히 생성되지 않음
- CAD의 벽 라인과 Revit에서 생성된 벽의 위치가 어긋남

### 원인
- 벽체 생성 시 위치 기준선(Location Line) 파라미터가 벽 중심선 기준으로 설정되지 않아 벽의 생성 방향과 위치가 CAD 원본과 불일치

### 수정 내용
- C# Revit 애드인에서 벽 생성 시 `WALL_KEY_REF_PARAM` 파라미터를 벽 중심선(Wall Centerline) 기준으로 설정
- 이를 통해 CAD 라인의 중심을 기준으로 벽체가 생성되어 시계방향 정방향이 Revit과 동일하게 적용됨
- 수정 파일: `CADLineWallData.cs`

- 기존 "+ 새 벽체 타입 추가" 방식은 프로젝트 초기에 수십 개의 벽체 타입을 등록해야 할 때 시간 소요가 큼

### 원인
- 기존 벽체 타입 관리가 개별 폼 입력 방식으로만 구현되어 일괄 처리가 불가능한 구조였음

- **병합/대체 모드**: 기존 데이터에 추가(병합) 또는 전체 대체 선택 가능
- **인라인 편집**: 테이블 셀 클릭으로 자재 선택, 우클릭으로 초기화, 드래그로 순서 변경
- **자동 계산**: 레이어별 두께, 자재비, 노무비, 합계 자동 집계
- **IndexedDB 저장**: `excelWallTypes` 스토어에 데이터 영구 저장

## 33. Revit에 동일한 벽체 타입 존재 시 중복 생성 방지

| 항목 | 내용 |
|------|------|
| **상태** | 완료 (접수→가능→완료) |
| **피드백** | 레빗에서 벽 작성을 먼저한 경우 → 벽두께가 다를때도 생성은 되나 → 경고창이 뜨면 어떨지? → 동일한 타입이 있으면 생성 안됨. 당연히 경고창도 뜸 |

### 증상
- Revit에 이미 동일한 이름의 벽체 타입이 존재할 때, Kiyeno에서 벽체 타입을 생성하면 중복 생성될 수 있음
- 두께가 다르더라도 같은 이름의 타입이 생성되어 혼란 발생 가능

### 원인
- 벽체 타입 생성 시 Revit에 동일 이름의 기존 타입이 있는지 확인하는 중복 검사 로직이 없었음

### 수정 내용
- Revit에서 벽체 타입 생성 전 동일 이름의 `WallType`이 이미 존재하는지 확인하는 중복 검사 추가
- 동일한 타입이 존재하면 생성을 중단하고 경고창(TaskDialog)으로 사용자에게 알림
- 두께가 다르더라도 이름이 같으면 중복으로 판단하여 생성 방지
- 수정 파일: `WallTypeCreate.cs`
